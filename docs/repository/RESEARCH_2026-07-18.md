# Repository Governance Research and Live Baseline

Research performed 2026-07-18 using current Git/GitHub primary documentation,
the GitHub REST API, `gh`, and local Git evidence. Web facts are live as of that
date; design recommendations are identified as conclusions.

## Executive conclusion

Use a protected-trunk GitHub Flow adapted for a solo human owner and several AI
execution lanes. The draft `feature/*`, `review/*`, `hotfix/*`, and
`experiment/*` strategy adds branch taxonomy without resolving ownership or
publication authority. Host-scoped short-lived branches plus PR metadata make
ownership explicit and keep `main` the only truth.

Do not perform a big-bang code-tree move. Nero's Constitution requires a
strangler-fig migration and current evidence shows split Git history is the
urgent structural risk. First reconcile history, install enforcement, and make
the existing modular boundaries measurable. Then migrate application packages
incrementally only where tests prove a benefit.

## Live GitHub baseline

| Control | Observed state | Risk |
|---|---|---|
| Visibility | Public | Changes and mistakes are externally visible |
| Default branch | `claude/github-repo-verification-vor5iw` | A task branch is presented as canonical |
| Rulesets | None | No central branch policy |
| `main` protection | None | Direct/force/delete updates are not remotely blocked |
| Required CI | None | PR #16 had no status checks |
| Merge methods | merge, squash, and rebase all enabled | History shape is inconsistent |
| Delete branch on merge | Disabled | Branch debt accumulates |
| Actions | All actions allowed; SHA pinning not required | Supply-chain surface is broader than needed |
| Secret scanning | Enabled; push protection enabled | Good current control |
| Dependabot security updates | Disabled | Known dependency fixes are not proposed automatically |
| Private vulnerability reporting | Disabled | No safe repository-native disclosure path |
| Collaborators | Toni only | A required peer approval count of one would deadlock self-authored PRs |

## Local baseline

- Active branch: `codex/ORCHESTRAION`, no upstream, legacy uppercase/typo name.
- Dirty baseline before governance edits: generated
  `.codex/environments/environment.toml`; preserved and not absorbed silently.
- School START logging then modified `School/DEBATE CC/LOG.txt` as required.
- `origin/main...HEAD`: 10 commits on `origin/main` absent locally and 34 local
  commits absent from `origin/main`.
- `origin/main` and the GitHub-default branch each had two unique commits.
- `origin/claude/m1-verifier-gate-repair` contains both remote lines.
- One registered worktree is prunable because its gitdir target is missing.
- System Python lacked runtime dependencies; the project `.venv` ran 62
  unittest-discovered tests green. The suite mixes unittest classes and
  pytest-style functions, so the new CI installs pytest to execute both forms.

Merge previews were performed with `git merge-tree`, without changing refs:

- `origin/main + HEAD`: four conflict paths.
- GitHub-default branch + `HEAD`: the same four conflict paths.
- `origin/main + origin/claude/m1-verifier-gate-repair`: clean.
- `origin/claude/m1-verifier-gate-repair + HEAD`: conflicts across 21 paths.

Conclusion: the M1 repair line is the clean remote convergence base, but its
merge with the latest recovery line requires explicit semantic resolution.

## Primary-source findings applied

1. GitHub Flow uses short-lived branches, PRs, checks, review, merge, and branch
   deletion. Applied as one canonical `main` plus host-scoped branches.
2. Git documents `pull --ff-only` as the non-divergence path. Applied only to a
   clean `main`; task branches fetch and reconcile explicitly.
3. Git worktrees isolate indexes and HEADs while sharing refs. Applied as one
   branch per worktree, with stale metadata treated as reviewable debt.
4. GitHub rulesets can require PRs, linear history, status checks, conversation
   resolution, and block deletion/force pushes. Applied in a disabled template
   with no bypass actors.
5. GitHub documents ruleset Evaluate mode as Enterprise-only. The personal
   public repo must review a disabled import, then explicitly activate it.
6. GitHub Actions security guidance recommends least-privilege tokens, avoiding
   privileged untrusted-code triggers, and pinning actions to full commit SHAs.
   All added actions are pinned to live-verified upstream commits and no
   `pull_request_target` trigger is used.
7. GitHub dependency review is available to public repositories and can block
   newly introduced high-severity vulnerabilities. Applied as a required PR
   check; license blocking is deferred because the optional voice-effects
   dependency has an explicitly documented GPL decision.
8. Artifact attestations establish build provenance but do not prove security.
   They are deferred until Nero publishes executable release artifacts.

## Alternatives rejected

- **Git Flow with permanent develop/release branches:** too much state for a
  solo maintainer and increases agent drift.
- **Direct-to-main with local hooks only:** hooks are bypassable and do not
  protect GitHub.
- **One required approving review now:** impossible with one collaborator for
  self-authored work; Toni's merge confirmation is the real human gate.
- **Merge queue now:** valuable at higher PR concurrency, but unnecessary before
  basic CI and a canonical trunk exist.
- **Mandatory signed commits now:** useful later, but would block current hosted
  lanes until identities and signing keys are deliberately provisioned.
- **Immediate `src/` rewrite:** violates the constitutional strangler-fig rule
  and compounds the unresolved branch divergence.
