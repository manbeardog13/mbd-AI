# ADR-0028: Canonical knowledge base and documentation hierarchy

**Status:** Proposed (Toni's acceptance promotes; drafted by Claude lane)
**Date:** 2026-07-16

## Context

The repo grew to 206 markdown files across four days of intense work. The
2026-07-16 audit (`docs/canon/AUDIT_REPORT.md`) found: no master index, no
onboarding path, superseded documents still speaking with authority (DIRECTIVE
vs Constitution priority order), capsule drift between repo canon (V1) and the
deployed Claude lane (V2), two colliding phase-numbering schemes, and one
load-bearing dependency (Proxima) documented nowhere in the repo. The guiding
requirement: **a new model must be able to understand Nero entirely from the
repo alone.**

## Decision

Adopt a canonical knowledge system with a fixed documentation hierarchy:

**Vision → Principles → ADRs → Specs → Implementation → Tests → Skills**

1. Three layers — **core / operational / archival** — declared per file in
   YAML frontmatter, with placement rules in
   `docs/canon/CANONICAL_STRUCTURE.md`.
2. A metadata, index, and changelog standard in
   `docs/canon/KNOWLEDGE_STANDARD.md`. The master index
   (`docs/canon/INDEX.md`) is generated by `scripts/build_canon_index.py`,
   never hand-edited.
3. Decisions stay in `docs/adr/` (why-shaped, ≤1 page); implementation
   contracts live in `docs/specs/*.spec.md` (what-shaped). A spec cites its
   ADRs; nothing below a level contradicts the level above.
4. Superseded documents carry a banner and `superseded_by` frontmatter; they
   are archived to `docs/archive/`, never silently deleted.
5. Onboarding contract: capsule → Constitution → `docs/canon/README.md` →
   INDEX → PROJECT_BRIEF, then cold reads on demand.
6. Migration follows `docs/canon/MIGRATION_PLAN.md`: additive layer executed
   2026-07-16; moves/edits phased, reference-checked, one commit per phase,
   gated on the rescue branch resolving.

## Consequences

- Onboarding any compatible model becomes a read-order, not archaeology.
- Index generation and (future) `verify_canon.py` turn documentation
  discipline into a mechanism instead of vigilance.
- Cost: frontmatter maintenance on knowledge files, and a migration debt
  (Phases A–E) that must actually be paid.
- School adopts the standard only through its own shared-work protocol.

## Alternatives considered

- **Wiki/external knowledge base:** rejected — violates local-first and
  repo-as-truth; splits canon from code and verify scripts.
- **Everything as ADRs:** rejected — ADRs bloat into unmaintainable
  pseudo-specs; decisions and contracts have different lifecycles.
- **Do nothing / rely on file naming:** rejected — the audit's drift findings
  are the empirical refutation.
